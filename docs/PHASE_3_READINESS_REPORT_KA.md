## რა გავაკეთეთ ამ ერთ სამუშაო დღეში

### 1. შემოწმდა „ნამდვილი მდგომარეობა"

თავიდან ვფიქრობდით, რომ Phase 2.5 (სწრაფი გამყარების ეტაპი) ჯერ
დაუმთავრებელია. რეალურად — დამთავრებულია. სამივე ქვე-ეტაპი:

- **2.5B** — საკითხავი მასალის სკალირება ✅ მუშაობს
  (326 სტატია, 5301 პატარა ცალი, 5302 ვექტორი, 568 ცნება)
- **2.5C** — ოჯახისთვის ხილული გვერდი ✅ მუშაობს
  (dashboard, ყოველდღიური დაიჯესტი, გადაუდებელი მუშტარი)
- **2.5D** — ჰიპოთეზების შემოწმების ნაკადი ✅ მუშაობს
  (5 დადასტურებული ჰიპოთეზა, 10 სასწავლო მაგალითი, ყველა ციტირებით)

ანუ ის რასაც ვაშენებდით — **უკვე აშენებულია, უბრალოდ დოკუმენტაცია ჩამორჩა.**

### 2. დოკუმენტაცია მოვაყვანე რეალობას

`CLAUDE.md`, `README.md`, `.planning/ROADMAP.md` ახლა შეესაბამება იმას, რაც
ნამდვილად დაშენდა. Phase 0, 1, 2, 2.5 აღნიშნული აქვს როგორც **დახურული**.
Phase 3 — როგორც **მზადაა დასაწყებად**.

ძველი აუდიტი (`PHASE_2_LIVE_AUDIT.md`, `ACTIVITY_DIAGNOSTIC_PLAN.md`) გადატანილია
`docs/archive/`-ში, რომ აღარ ჩაგერიოდეს თვალში, მაგრამ ისტორიად დარჩა.

### 3. დაემატა ოჯახის თვალით სანახავი 3 ახალი გვერდი

`viewer/app/`-ში გაჩნდა:

- `/papers` — სტატიების სია, რომელიც აქამდე ჩავიკითხეთ
- `/therapies` — მკურნალობების სტატუსი (აქტიური, დაგეგმილი, განხილვის ქვეშ)
- `/timeline` — ალექსანდრას მნიშვნელოვანი მოვლენების სია თარიღების მიხედვით

ეს გვერდები **არ ცვლის მონაცემებს, მხოლოდ კითხულობს.** არც სტილი, არც
ანიმაცია, არც „ლამაზი UX" — უბრალოდ მონაცემები ეკრანზე. **არც MRI არ ერთვება
ამ გვერდებს** (ის მხოლოდ ბრაუზერშია, არასოდეს — სერვერზე).

ასევე dashboard-ისა და hypotheses გვერდების ნავიგაცია გავაუმჯობესე — ახლა
ყველა გვერდს აქვს მისახვედრი ლინკები ერთმანეთთან.

### 4. ბიუჯეტის გამოსაყენებელი წყაროს გასწორება

n8n-ის ბიუჯეტ-კარიბჭე ადრე ითვლიდა მხოლოდ ორი ტიპის ხარჯვას (`agent_run` და
`fire_drill`). მაგრამ Anthropic-ის ყოველი ჩახმობა იწერებოდა როგორც
`llm_call`. ანუ ნამდვილი ხარჯი მცირდებოდა მონიტორინგში.

ფაილი (`workflows/daily-budget-gate.json`) უკვე გასწორებული აქვს მესამე ტიპს
(`llm_call`). მაგრამ ეს **მხოლოდ კოდშია** — ფაილი n8n-ის სერვერზე ხელახლა
უნდა გადატვირთონ, რომ რეალურმა workflow-მა აიღოს ცვლილება. ეს არის
**ერთადერთი ღია ოპერაციული საკითხი**.

წარმოიდგინე, რომ ალექსანდრას სამედიცინო ბიბლიოთეკას ვაშენებთ:

**მკვლევარის შესახვედრად ბიბლიოთეკა საკმარისად სავსეა.**

## რა გვაქვს ხელში Phase 3-სთვის

Phase 3 ააქცევს ოთხ ახალ შესაძლებლობას:

1. **Communicator გააქტიურება** — სისტემა ფარდულის სახით მოამზადებს ოჯახური
   შეტყობინებების „დრაფტებს". ყოველი დრაფტი წყაროზე მიმთითებელი. ყოველი
   დრაფტი ჯერ **ჩვენ შევთანხმდებით**, შემდეგ გაიგზავნება.

2. **გაფრთხილების დონეები** — T0 (დაბლოკილი), T1 (გადაუდებელი), T2 (მოქმედება საჭიროა),
   T3 (მნიშვნელოვანი), T4 (კვირეული). „გადაუდებლად" მხოლოდ ისეთი რამ
   მიდის რომელიც *მაღალი ნდობით* და *დროზე დამოკიდებულია*. სხვა ყველაფერი —
   ან კვირეულ ბრიფში, ან ყოველდღიურ დაიჯესტში.

3. **მკვლევარების/კლინიცისტებისთვის ემაილის გაგზავნა** — ისევ მხოლოდ
   **დრაფტი**. ბოლო ღილაკი ყოველთვის შენი. დღეში მაქსიმუმ 5 ემაილი.

4. **კვირეული PDF რეპორტი** — 1 გვერდიანი რეზიუმე + სრული ციტირების
   დანართი. ცარიელი სექციები იქნება „ამ კვირეში ახალი არ არის",
   რომ ცარიელად არ გაუჩინარდეს.

ყველაფერი ერთ წესს ემორჩილება:

> **ყოველი წინადადება წყაროზე მიმთითებელია.**
> **ყოველი გადაწყვეტილება — ექიმის ხელშია.**
> **ყოველი გაგზავნა — ადამიანი ადასტურებს.**

### 1. n8n ბიუჯეტ-კარიბჭე — გასაშვები ხელახლა

  ნამდვილია. **მაგრამ კოდის მხარეს** (Python-ში) ბიუჯეტის შემოწმება
  მუშაობს — ანუ ბიუჯეტი ფაქტიურად დაცულია.

  hypotheses, contacts, ...) აქვს RLS ჩართული, მაგრამ წესი ცარიელია
  (`USING (true)` ყველასთვის).

  ეს ცხრილები. **MRI არ ჩანს**, არც ფაქტობრივი ცხოვრების მონაცემები.
  ყველაზე ფაქიზი ნაწილია — კონტაქტების სია (ექიმების ემაილები).
- **რა გაკეთდება:** Phase 3-ში პირველ კვირას — მცირე მიგრაცია (`008_*.sql`)
  რომელიც წესებს მკაცრად მიამაგრებს `service_role`-ს და `authenticated`-ს.
  დაწერილია `docs/SECURITY_AUDIT.md`-ში.

### 3. ფრონტენდის გვერდები ჯერ მინიმალური ფორმითაა
- **რა:** /papers, /therapies, /timeline მუშაობს, მაგრამ უსტილოა,
