## 1. მონაცემთა ბაზები

### რატომ ვიწყებთ ბაზებით

სანამ სისტემა რაიმეს მოძებნის, მას სჭირდება ადგილი სადაც ნაპოვნს შეინახავს. ჩვენი სისტემა სამ სხვადასხვა ტიპის ინფორმაციას იყენებს, ამიტომ სამი სხვადასხვა საცავი გვჭირდება. ეს არ არის სიჭარბე — თითოეული იმას აკეთებს რაც მეორეს არ შეუძლია.

### რა უნდა გაკეთდეს

**Supabase** არის ცხრილების ბაზა — კონტაქტები, ვადები, სტატიების ქულები, ლოგები. ყველაფერი რაც ცხრილში ეტევა. ამას ვქმნით პირველს რადგან დანარჩენი ორი ბაზა Supabase-ზეა დამოკიდებული metadata-სთვის. შევქმნით 10 ცხრილს: papers, contacts, therapies, brain_regions, hypotheses, clinical_trials, relationships, discovery_reports, aleksandra_timeline, ingestion_log. სქემა უკვე გვაქვს მზა SQL ფაილის სახით.

→ *იხილეთ ჰენდაუთი, პუნქტი 1.1*

**Neo4j** არის ცოდნის რუკა — კავშირები ენტიტეტებს შორის. „წამალი X მოქმედებს pathway Y-ზე, pathway Y აქტიურია ტვინის რეგიონ Z-ში, რეგიონი Z დაზიანებულია ალექსანდრას ტვინში" — ეს ჯაჭვი Neo4j-ში ერთი query-ით მოიძებნება. Graphiti ბიბლიოთეკას ვიყენებთ რადგან ის ავტომატურად ამოიცნობს ენტიტეტებს ტექსტიდან და ტემპორალურ წონას ანიჭებს (ძველი კვლევა = ნაკლები წონა). შევქმნით 9 ენტიტეტის ტიპს და 9 კავშირის ტიპს.

→ *იხილეთ ჰენდაუთი, პუნქტი 1.2*

**Qdrant** არის ასოციაციური მეხსიერება — „იპოვე სტატია რომელიც ამ სტატიას ჰგავს." ყოველი სტატიის ტექსტი ციფრულ „თითის ანაბეჭდად" (embedding) გარდაიქმნება და Qdrant-ში ინახება. embedding-ების შესაქმნელად fastembed-ს ვიყენებთ რომელიც ლოკალურად მუშაობს — არანაირი მონაცემი ღრუბლოვან სერვისს არ ეგზავნება, რაც პაციენტის მონაცემთა პრივატულობისთვის კრიტიკულია. შევქმნით 3 collection-ს: papers, therapies, hypotheses.

→ *იხილეთ ჰენდაუთი, პუნქტი 1.3*

**Cloudflare R2 და KV** არის დამხმარე საცავი. R2-ში ვინახავთ დიდ ფაილებს (PDF-ები, მომავალში MRI-ები და 3D mesh-ები). KV-ში ვინახავთ ხშირად გამოყენებულ მონაცემებს (dashboard-ის სტატისტიკა) სწრაფი წვდომისთვის.

→ *იხილეთ ჰენდაუთი, პუნქტი 1.4*

## 2. ავტომატიზაციის ძრავი

### რატომ გვჭირდება ავტომატიზაცია ფუნდამენტშივე

სისტემის მთავარი ღირებულება ისაა რომ ის თავისით მუშაობს — ყოველ 6 საათში, ადამიანის ჩარევის გარეშე. ეს ავტომატიზაცია n8n-ით ხდება. n8n-ის გაშვება ფუნდამენტშივე არის საჭირო რადგან მომდევნო ეტაპზე Spider Agent-ს n8n-ის cron trigger ამოქმედებს.

### რა უნდა გაკეთდეს

**n8n** არის workflow ავტომატიზაციის პლატფორმა. Railway-ზე ვდეპლოიებთ (ღრუბლოვანი სერვერი, $5/თვე). პირველ ეტაპზე ერთ test workflow-ს შევქმნით: ყოველ 6 საათში ეშვება, test endpoint-ს მიმართავს, შედეგს Telegram-ში აგზავნის. ეს ჩონჩხია — მომდევნო ეტაპზე test endpoint-ის ნაცვლად PubMed API იქნება.

→ *იხილეთ ჰენდაუთი, პუნქტი 2.1*

**Telegram Bot** არის არხი სისტემასა და ოჯახს შორის. BotFather-ით ვქმნით ბოტს, ვამატებთ ჯგუფში, და ვაკონფიგურირებთ სამ დონის შეტყობინებას: 🔴 სასწრაფო (ახალი trial, safety alert — მყისიერად), 🟡 მნიშვნელოვანი (კარგი სტატია — დღეში ერთხელ), 🟢 ყოველკვირეული (ანგარიში — კვირა 09:00). ეს alert tiers სისტემის მთელი ცხოვრების განმავლობაში უცვლელი რჩება.

→ *იხილეთ ჰენდაუთი, პუნქტი 2.2*

### რა უნდა გაკეთდეს

**Claude API** არის ხელოვნური ინტელექტის ძრავი რომელიც ყველა აგენტს აძლიერებს. API key-ს ვიღებთ console.anthropic.com-ზე და .env ფაილში ვინახავთ.

→ *იხილეთ ჰენდაუთი, პუნქტი 3.1*

**CrewAI** არის 5 აგენტის ორკესტრატორი. თითოეულ აგენტს ვანიჭებთ: role (თანამდებობა), goal (მიზანი), backstory (კონტექსტი). Tools ცარიელია — ინსტრუმენტებს მიმართულება I-III-ში მივცემთ. ფუნდამენტში მხოლოდ ჩონჩხს ვქმნით და ვამოწმებთ რომ 5-ვე ინიციალიზდება.

**mem0** არის აგენტების საერთო მეხსიერება. როცა Spider პოულობს სტატიას, Analyzer-მა უნდა იცოდეს რომ ეს სტატია უკვე ნაპოვნია. mem0 Qdrant-ს იყენებს backend-ად (უკვე გაშვებული გვაქვს) და 5 აგენტისთვის ქმნის საერთო მეხსიერების სივრცეს.

## 5. MCP სერვერების ინსტალაცია

### რატომ არის MCP ცენტრალური

MCP (Model Context Protocol) არის სტანდარტი რომლითაც Claude-ს გარე ინსტრუმენტებს ვაძლევთ. ALEKSANDRA_BRAIN-ის 52 MCP სერვერი = 52 ინსტრუმენტი Claude-ისთვის. ფუნდამენტში ვრწმუნდებით რომ ყველა მუშაობს.

### რა უნდა გაკეთდეს

**FastMCP** არის ჩვენი custom MCP სერვერების ასაგებად. Python-ის decorator-ებით 10 ხაზ კოდში ვქმნით MCP სერვერს. ფუნდამენტში ვაინსტალირებთ და hello-world სერვერს ვქმნით — მომდევნო ეტაპებში 5 world-first სერვერი ამ ფრეიმვორკით აიგება.

→ *იხილეთ ჰენდაუთი, პუნქტი 5.1*

**Claude.ai Registry MCP-ები (23)** უკვე დაკავშირებულია Claude.ai-ში. ფუნდამენტში ვამოწმებთ თითოეულს: PubMed ეძებს? Gmail კითხულობს? Calendar ხედავს? Clinical Trials პასუხობს? თუ რომელიმე გათიშულია — reconnect.

→ *იხილეთ ჰენდაუთი, პუნქტი 5.2*

**GitHub Self-hosted MCP-ები (19)** ცალ-ცალკე ინსტალაციას საჭიროებს. 5 პრიორიტეტულს (Graphiti, Qdrant, Supabase, Firecrawl, n8n-MCP) ჯერ ვაყენებთ და ვტესტავთ, შემდეგ დანარჩენ 14-ს.

→ *იხილეთ ჰენდაუთი, პუნქტი 5.3*

**AI Pulse Georgia MCP-ები (5)** ახალი ეკოსისტემიდან: FastMCP (უკვე დაყენებული 5.1-ში), Crawl4AI RAG, Prism MCP (HIPAA), Draw.io MCP, Perplexity MCP.

→ *იხილეთ ჰენდაუთი, პუნქტი 5.4*

## 6. მონაცემების მიგრაცია

### რატომ ვმიგრირებთ არსებულ მონაცემებს

ALEKSANDRA_BRAIN ნულიდან არ იწყება — 10 თვის ადვოკატირების კამპანიაში 60+ სტატია, 80+ კონტაქტი, სამედიცინო ისტორია და მკურნალობის pipeline უკვე არსებობს. ეს ინფორმაცია ახალ ბაზებში უნდა გადავიდეს რომ სისტემამ პირველივე დღიდან იცოდეს ყველაფერი რაც ჩვენ ვიცით.

### რა უნდა გაკეთდეს

**60+ სამეცნიერო სტატია** project knowledge-დან გადავა სამივე ბაზაში: Supabase (metadata + scores), Qdrant (embeddings სემანტიკური ძიებისთვის), Neo4j (entities + relationships). migration script batch-ით ამუშავებს: PDF-იდან ამოიღებს title, authors, year, DOI, abstract-ს, relevance score-ს ანიჭებს, და სამივე ბაზაში ჩაწერს.

→ *იხილეთ ჰენდაუთი, პუნქტი 6.1*

**80+ კონტაქტი** — ექიმები, მკვლევარები, კოორდინატორები, ფონდები — Supabase-ში და Neo4j-ში გადავა. Neo4j-ში კავშირები აისახება: Contact -[:WORKS_AT]→ Institution, Contact -[:MANAGES]→ Trial. follow-up ვადები Supabase-ში ინახება daily alert-ისთვის.

→ *იხილეთ ჰენდაუთი, პუნქტი 6.2*

**ალექსანდრას ქრონოლოგია** (28.08.2025 → დღემდე), აქტიური თერაპიები (Keppra, Vigabatrin, Duke EAP planned, Wisconsin A2 active), და ტვინის რეგიონების damage map (14+ რეგიონი) — ყველაფერი Supabase-ის შესაბამის ცხრილებში.

→ *იხილეთ ჰენდაუთი, პუნქტი 6.3*

## 7. უსაფრთხოება

### რატომ არის უსაფრთხოება ფუნდამენტის ნაწილი

ალექსანდრას სამედიცინო მონაცემები სენსიტიურია. უსაფრთხოება არ არის „მოგვიანებით დავამატებთ" — ის ფუნდამენტშივე უნდა იყოს ჩაშენებული. ერთი შეცდომა (API key GitHub-ზე, MRI cloud-ში) შეუქცევადია.
