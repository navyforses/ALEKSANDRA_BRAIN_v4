## P2 Incident — compose_bilingual misuse

**Root cause.** Plan 06.1-03 specified `compose_bilingual(prompt=en_text, client=…)` as the translation primitive for the backfill. `compose_bilingual` is a **DRAFTING tool** (Anthropic strict tool_use, Phase 6 plan 06-09): given a prompt, it COMPOSES new bilingual content from scratch. Used on a 30-char title like "Vigabatrin for Seizure Control", it generated 1–3 KB multi-paragraph clinical dossiers and frequently left ka empty (55/62 rows).

**Effect.** 62/62 eligible rows OVERWRITTEN with dossier-en + blank-ka. Original short titles LOST in the en field of the affected rows.

**Recovery options considered.**
- **Supabase backup restore** (most professional): rejected — Supabase Free tier dashboard confirmed "LAST BACKUP: No backups". No PITR available on Free.
- **Partial header-extraction recovery** (chosen): regex-extract first `**bold**` header from corrupted dossier en (compose_bilingual typically echoed the input as the first markdown header). ~89% effective on title fields.

**Spend.** 013 wasted run ≈ $1.50; 014 recovery ≈ $0.50; 015 retry ≈ $0.10. **Total Phase 6.1 spend ≈ $2.10** (cumulative project spend ~$7-8 / $60 cap).

**Lesson for future bilingual content work.** `compose_bilingual` is for DRAFTING new content (e.g., a weekly_brief section from raw data); it is NOT a translation primitive. For translating existing en → ka, use direct `anthropic.messages.create()` with a translator system prompt (the pattern in 014/015). Plan 06.1-03 should have specified this primitive distinction explicitly; the planner / executor missed the semantic difference because both tools live in `scripts/communicator/bilingual.py`.

**Mitigation (future).** Future bilingual translation work should:
1. Use the 014/015 translator pattern (direct Anthropic call with translator system prompt), not compose_bilingual.
2. Prefer sonnet-4-6 over sonnet-4-5 for medical/scientific content (less aggressive safety classifier).
3. Always wrap content access in `[b for b in resp.content if b.type == 'text']` defensively — refusals return `content=[]` with `stop_reason='refusal'`, NOT an exception.
