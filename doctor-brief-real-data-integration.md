# Doctor Brief Builder — real-data integration rule

**Author:** Manus AI

ეს დოკუმენტი განსაზღვრავს, როგორ უნდა დაემატოს Doctor Brief Builder არსებულ `ALEKSANDRA_BRAIN_v4/viewer` პორტალში ისე, რომ საიტზე არ გამოჩნდეს არც ერთი მოგონილი, illustrative ან placeholder მონაცემი. ფუნქცია უნდა იყოს არსებული portal shell-ის ნაწილი და არა დამოუკიდებელი საიტი, landing page ან პარალელური მონაცემთა სისტემა.

> ძირითადი წესი: თუ კონკრეტულ თემაზე source-backed ჩანაწერი არ არსებობს, Doctor Brief Builder-მა არ უნდა შექმნას ტექსტი ვარაუდით. ასეთ შემთხვევაში შესაბამის ბლოკში უნდა გამოჩნდეს „მონაცემი არ არის“.

| სფერო | დასაშვები ქცევა | აკრძალული ქცევა |
|---|---|---|
| მონაცემის წყარო | მხოლოდ არსებულ structured content/data layer-ში არსებული ჩანაწერის ჩვენება | ხელით ჩაწერილი სტატისტიკის, ზოგადი claim-ის ან demo ტექსტის ჩვენება როგორც ფაქტი |
| brief-ის ტექსტი | არსებულ evidence/risk/question ჩანაწერებზე დაყრდნობილი კომპოზიცია | ახალი სამედიცინო დასკვნის, რეკომენდაციის ან წყაროს გარეშე შეჯამების გენერაცია |
| no-data მდგომარეობა | მკაფიო ტექსტი: „მონაცემი არ არის“ | ცარიელი სექციის დამალვა ისე, თითქოს მონაცემი უბრალოდ გამოტოვებულია |
| UI ინტეგრაცია | არსებული `/ka/resources` ან portal flow-ის შიგნით ჩაშენება | ცალკე `/brief`, standalone ან marketing-style გვერდის შექმნა |
| ენა | ქართული primary ტექსტი და საჭიროების შემთხვევაში ინგლისური secondary ტექსტი | მხოლოდ ინგლისური სისტემური აღწერები ქართულ route-ზე |
| უსაფრთხოება | ექიმთან საუბრისთვის მოსამზადებელი research brief | სამედიცინო რჩევა, მკურნალობის ინსტრუქცია ან დიაგნოზის ჩანაცვლება |

Doctor Brief Builder-ის სწორი მოდელი არის read-only composition layer. იგი არ უნდა იყოს ახალი ცოდნის გენერატორი; ის უნდა იყოს არსებული, შემოწმებადი ჩანაწერების გამაერთიანებელი. თუ ჩანაწერი არ არსებობს, UI-მ ეს პირდაპირ უნდა თქვას და მომხმარებელს არ უნდა შეუქმნას შთაბეჭდილება, რომ მონაცემი არსებობს.

## ინტეგრაციის რეკომენდებული ნაკადი

პირველ ეტაპზე `/ka/resources` გვერდზე უნდა დაემატოს brief preview ბლოკი, რომელიც კითხულობს იმავე topic model-ს, რომელსაც PortalContent უკვე იყენებს. ყოველი სექცია უნდა დაიყოს სამ ნაწილად: **რა მონაცემი არსებობს**, **რა არის შეზღუდვა**, და **რა კითხვა შეიძლება დაისვას ექიმთან**. ამ სამი ბლოკიდან თითოეული უნდა მუშაობდეს დამოუკიდებელი no-data fallback-ით.

| brief სექცია | required input | input-ის არქონისას |
|---|---|---|
| Summary | source-backed topic summary | „მონაცემი არ არის“ |
| Evidence | მინიმუმ ერთი წყაროიანი evidence item | „მონაცემი არ არის“ |
| Risk / limitation | source-backed limitation ან uncertainty | „მონაცემი არ არის“ |
| Doctor questions | არსებული topic question ჩანაწერი | „მონაცემი არ არის“ |
| References | წყაროს URL/იდენტიფიკატორი/metadata | „მონაცემი არ არის“ |

ამ წესით brief builder არ ქმნის დამოუკიდებელ შინაარსს. იგი მხოლოდ აწყობს იმას, რაც მონაცემთა ფენაში უკვე არსებობს. ამიტომ თუ მონაცემთა ფენა ცარიელია, საბოლოო brief უნდა იყოს მოკლე და გამჭვირვალე: გამოჩნდეს title, disclaimer და სექციებში „მონაცემი არ არის“.

## implementation guardrails

კოდში უნდა იყოს ერთი საერთო helper, რომელიც ცარიელ string-ს, ცარიელ array-ს, `null`-ს და `undefined`-ს ერთნაირად ექცევა როგორც no-data მდგომარეობას. UI-ში hardcoded მაგალითები, პროცენტები, source count-ები ან clinical claim-ები არ უნდა დარჩეს. სარედაქციო/ინსტრუქციული ტექსტი დასაშვებია მხოლოდ მაშინ, თუ იგი ნათლად აღწერს პორტალის ფუნქციას და არ გამოიყურება როგორც მონაცემი.

მომდევნო ტექნიკური ნაბიჯი არის build/lint ვალიდაცია და smoke test იმ route-ებზე, სადაც no-data მდგომარეობა უნდა გამოჩნდეს. თუ ყველა route სუფთად აიწყობა, ცვლილებები უნდა დაკომიტდეს და production-ზე შემოწმდეს.
